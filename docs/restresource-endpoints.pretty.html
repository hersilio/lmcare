2.4 Listar appointments do cliente

Endpoint pretendido:

POST /services/apexrest/appointments/v2/client/{clientMdmId}

Request JSON:

{}

Response JSON de sucesso:

{
  "totalSize": 2,
  "records": [
    {
      "Id": "08pXXXXXXXXXXXXXXX",
      "AppointmentNumber": "SA-00012345",
      "CreatedDate": "2026-03-18T12:00:00.000Z",
      "Status": "Scheduled",
      "SchedStartTime": "2026-03-20T09:00:00.000Z",
      "SchedEndTime": "2026-03-20T10:00:00.000Z",
      "Description": "Agendamento via Portal",
      "ParentRecordId": "00QXXXXXXXXXXXXXXX"
    }
  ]
}

Response JSON de erro pretendido:

{
  "error": {
    "code": "Not Found",
    "message": "No appointments found for the client.",
    "requestId": null
  }
}

Observações importantes:

  • Embora o comentário do código sugira PATCH e CANCEL, a classe expõe somente @HttpPost.
  • O roteamento interno usa requestURI.split('/')[3] e split('/')[4], o que não bate com a URI completa padrão do Apex REST (/services/apexrest/...). Na prática, a rota pode não cair nos when 'appointment' e when 'client' como esperado.
  • O código da resposta tem problemas de inicialização:
  • AppointmentResponseDTO responseDTO; não é instanciado antes de responseDTO.result = resultDTO.
  • ErrorResponse também não instancia error antes de acessar error.code.
  • Para a equipe de engenharia, vale tratar a v2 como contrato pretendido, não como contrato hoje confiável em produção.

5. SCH_DailyAvailablityController

Classe: SCH_DailyAvailablityController

Endpoint:

GET /services/apexrest/dailyAvailability

Query params obrigatórios:

  • startDate no formato YYYY-MM-DD
  • endDate no formato YYYY-MM-DD
  • workTypeGroupId
  • storeId
  • engagementChannelTypeId

Request JSON:

null

GET /services/apexrest/dailyAvailability?startDate=2026-03-20&endDate=2026-03-31&workTypeGroupId=0VGAe0000000001&storeId=0HhAe0000001ENVKA2&engagementChannelTypeId=0eFAe000000034LMAQ
